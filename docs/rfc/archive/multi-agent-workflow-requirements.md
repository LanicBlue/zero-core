## 1. 背景与目标

### 1.1 背景

zero-core 已具备单 Agent 执行能力（AgentLoop、工具系统、Hook、MCP）以及基础的子 Agent 委派。但软件开发是一个**持续循环的过程**——需求发现、讨论确认、任务拆解、编码实现、质量审查、测试验证、归档沉淀——单 Agent 难以同时兼顾这些不同维度的持续工作。

### 1.2 设计哲学：Agent 自驱动

本系统的核心理念是 **Agent 自驱动，用户只做方向把控和细节补充**。

这意味着：
- Agent 主动巡检、主动发现问题、主动推进需求流转——不需要用户逐一触发
- 用户的角色是**决策者和纠偏者**：确认方向、补充细节、审核关键产出、纠正偏差
- 系统应该让用户"看到 Agent 在做什么"而非"告诉 Agent 该做什么"
- 理想状态下，用户打开 zero-core 看到的是：Agent 已经发现的问题清单、已经在推进的任务、已经完成等待验收的成果

### 1.3 目标

在 zero-core 现有 runtime 之上，构建一个 **Agent 自驱动的持续型软件开发生命周期系统**，核心特征：

- **自驱巡检**：项目分析 Agent 定期扫描项目，自动发现问题和改进点，维护项目 Wiki 和需求池
- **自驱流转**：Agent 主动推进需求状态流转（规划→执行→验证），用户仅在关键节点参与
- **分工协作**：不同角色的 Agent 各司其职，由任务 Lead 统一调度，自主推进
- **人在环路**：用户做方向把控和细节补充——确认需求、审核方案、验收成果
- **知识沉淀**：每次任务完成后自动更新项目 Wiki，形成持续积累

### 1.4 非目标

- 不做通用 AI Agent 编排平台（聚焦软件开发场景）
- 不做远程/分布式 Agent 执行（MVP 阶段仅本地执行）
- 不做 Agent 角色市场（预置角色为主，自定义角色为辅）

### 2.2 核心实体

| 实体 | 说明 |
|------|------|
| **Project（项目）** | 关联的项目目录，一个项目绑定一个项目分析 Agent 和一套 Wiki |
| **Requirement（需求）** | 最小交付单元，跨项目共享池，归属一个具体项目，有明确的生命周期状态 |
| **Project Wiki（代码知识树）** | 树形结构的项目知识库，浅层加载 + 按需展开节点 |
| **Agent Role（角色）** | 预置的 Agent persona，定义职责、工具、模型偏好 |
| **Orchestrator（编排器）** | Lead 的内置工具，负责创建/管理 sub-agent 实例，返回结果摘要 |
| **Execution Step（执行步骤）** | 一次 sub-agent 执行：编码、审查、测试等，由编排器调度 |

### 2.3 需求生命周期

这是整个系统的核心状态机：

```
                    ┌──────────────────────────────────────────────┐
                    │              项目分析 Agent (常驻)             │
                    │   定时巡检 → 发现问题/建议/优化 → 自动入池     │
                    └──────────────────┬───────────────────────────┘
                                       │
                                       ▼
                              ┌─────────────────┐
                    ┌────────▶│  💡 发现 (Found) │◀──── 用户手动创建
                    │         └────────┬────────┘
                    │                  │ 用户点击需求
                    │                  ▼
                    │         ┌─────────────────┐
                    │         │ 💬 讨论 (Discuss)│  用户与 Agent 对话
                    │         └────────┬────────┘
                    │                  │ 用户确认需求明确
                    │                  ▼
                    │         ┌─────────────────┐
                    │         │ ✅ 就绪 (Ready)  │  需求明确，可分配
                    │         └────────┬────────┘
                    │                  │ 任务 Lead 领取 & 规划
                    │                  ▼
                    │         ┌─────────────────┐
                    │         │ 📋 规划 (Plan)   │  Lead 拆解任务，等待审核
                    │         └────────┬────────┘
                    │                  │ 审核通过（项目分析/用户）
                    │                  ▼
                    │         ┌─────────────────┐
          驳回/打回  │         │ 🔨 执行 (Build)  │  Dev 编码 → Review → QA 测试
                    │         └────────┬────────┘
                    │                  │ 执行完成
                    │                  ▼
                    │         ┌─────────────────┐
                    │         │ 🔍 验证 (Verify) │  项目分析/用户验收
                    │         └────────┬────────┘
                    │                  │ 验收通过
                    │                  ▼
                    │         ┌─────────────────┐
                    └─────────│ 📦 归档 (Closed) │  更新 Wiki，关闭需求
                              └─────────────────┘
```

**状态说明**：

| 状态 | 触发者 | 描述 |
|------|--------|------|
| **Found（发现）** | Agent 自驱（项目分析巡检）或用户手动 | 新需求入池，等待用户关注 |
| **Discuss（讨论）** | 用户点击（方向把控） | 用户点击需求进入对话，Agent 引导讨论、细化需求 |
| **Ready（就绪）** | 用户确认（方向把控） | 讨论完毕，用户确认需求明确 |
| **Plan（规划）** | Agent 自驱（任务 Lead 自动领取） | Lead 拆解步骤、分配角色，等待审核 |
| **Build（执行）** | Agent 自驱（Lead 调度 Dev/Reviewer/QA） | 编码 → 审查 → 测试，自主循环直到质量达标 |
| **Verify（验证）** | Agent 自驱（项目分析验收）或用户 | 检查执行结果是否符合需求和设计 |
| **Closed（归档）** | Agent 自驱（验证通过自动归档） | 更新 Wiki，关闭需求 |

> **自驱动原则**：Agent 在每个节点主动推进，不等待用户指令。用户仅在 **确认需求**（Found→Ready）和 **审核规划**（Plan→Build）时参与；执行、验证、归档由 Agent 自主完成。

**异常流转**：
- Verify 验证不通过 → Agent 自驱回退到 Build 重新执行
- Build 过程中发现需求不明确 → Agent 自驱回退到 Discuss，请用户补充
- 用户随时可取消需求（任何状态 → Cancelled）——这是用户的方向纠偏手段

## 3. 预置角色定义

### 3.1 项目分析 (Project Analyst)

| 属性 | 值 |
|------|-----|
| **运行模式** | **常驻**，定时巡检 + 手动触发 |
| **核心职责** | ① 定期分析项目全局状态 ② 维护项目 Wiki ③ 发现问题/建议/优化方向，自动入需求池 ④ 任务完成后验收并更新 Wiki |
| **System Prompt 重点** | 你是项目的全知视角守护者。持续关注项目健康度、架构一致性、代码质量趋势、依赖安全性，以及功能演进方向。你的产出是结构化的需求建议和 Wiki 更新。 |
| **推荐模型** | 强推理模型（如 Claude Opus） |
| **可用工具** | 全部工具（Read/Write/Edit、Shell、Grep/Glob、Git、子 Agent） |
| **特殊能力** | 可读写 Wiki、可创建需求、可验收任务结果 |

**定时巡检内容**：
- 代码变更分析：自上次巡检以来的 diff，识别潜在问题
- 依赖检查：过时/有安全风险的依赖
- 测试覆盖率趋势
- 架构一致性检查：新增代码是否符合既有架构
- 文档同步检查：代码变更后文档是否需要更新
- 功能演进建议：基于项目方向和代码趋势，提出未来功能建议

### 3.2 任务 Lead (Task Lead)

| 属性 | 值 |
|------|-----|
| **运行模式** | 按需激活，有就绪需求时启动 |
| **核心职责** | ① 领取就绪需求 ② 拆解为步骤序列 ③ 通过编排器工具调度 sub-agent ④ 审核每步结果摘要 ⑤ 完成后提交审核 |
| **System Prompt 重点** | 你是任务执行的总指挥。拿到明确需求后，合理拆解为可执行步骤，通过 orchestrate() 工具分配给对应角色，把控执行质量，确保交付物符合需求定义。你不直接控制 sub-agent，而是通过编排器工具间接调度。 |
| **推荐模型** | 强推理模型（如 Claude Opus） |
| **可用工具** | 全部工具 + **编排器工具（orchestrate）** |
| **特殊能力** | 可通过编排器调度其他角色、可提交审核 |

**编排器工具 (orchestrate) 接口**：

```typescript
// Lead 调用编排器
orchestrate({
  role: 'developer' | 'reviewer' | 'qa',
  task: '具体任务描述',
  context: {
    requirement: { id, title, discussion_summary },
    wiki_nodes: ['src/payment/', 'src/payment/callback-handler.ts'],  // Lead 指定展开
    related_files: ['src/payment/wechat-pay.ts']
  }
})

// 编排器返回给 Lead 的是结果摘要（非完整输出）
// { changed_files: [...], summary: '...', issues: [...] }
```

**关键设计**：
- Lead 使用**同一个 AgentLoop 持续会话**，保持上下文连续性
- sub-agent（Dev/Reviewer/QA）每次步骤创建**新 AgentLoop 实例**
- Lead 不直接碰 sub-agent，编排器负责：创建实例 → 注入上下文（Wiki 节点展开 + 需求 + 任务指令）→ 执行 → 返回摘要
- Lead 的上下文只存摘要，不被 sub-agent 海量输出撑爆

### 3.3 代码实现 (Developer)

| 属性 | 值 |
|------|-----|
| **运行模式** | 由任务 Lead 调度 |
| **核心职责** | 根据任务 Lead 的指令生成或修改代码 |
| **System Prompt 重点** | 你是高效的代码实现者。专注于分配给你的任务，遵循项目代码规范，编写可测试、可维护的代码。 |
| **推荐模型** | 速度与能力均衡的模型（如 Claude Sonnet） |
| **可用工具** | 文件操作（Read/Write/Edit）、Shell、Grep/Glob、子 Agent |
| **特殊能力** | 无，仅执行分配的编码任务 |

### 3.4 质量审查 (Quality Reviewer)

| 属性 | 值 |
|------|-----|
| **运行模式** | 由任务 Lead 调度 |
| **核心职责** | 对特定代码和功能做质量审查 |
| **System Prompt 重点** | 你是严格的质量守门人。以批判性视角审查代码，关注正确性、安全性、性能、可维护性和最佳实践。给出具体可操作的改进建议。 |
| **推荐模型** | 强分析能力模型（如 Claude Sonnet/Opus） |
| **可用工具** | 文件操作（只读）、Grep/Glob、Git diff/log |
| **特殊能力** | 可标记审查通过/不通过，不通过时附带修改建议 |

### 3.5 测试验证 (QA Tester)

| 属性 | 值 |
|------|-----|
| **运行模式** | 由任务 Lead 调度 |
| **核心职责** | 编写和执行测试 |
| **System Prompt 重点** | 你是测试专家。关注边界条件、异常路径、测试覆盖率和可重现性。确保代码变更经过充分验证。 |
| **推荐模型** | 速度优先模型（如 Claude Haiku/Sonnet） |
| **可用工具** | 文件操作（Read/Write）、Shell（运行测试）、Grep/Glob |
| **特殊能力** | 可标记测试通过/失败，报告缺陷 |

- **底部操作**：手动添加需求、立即触发项目分析

│  明白了。确认需求：                                            │

│  - 支付回调增加幂等性处理                                      │

**说明**：

- 用户确认后，状态变为「就绪」
- 需求进入执行后，对话方切换为**任务 Lead**
- 对话历史随需求持久化，状态切换时可回溯

- Wiki 采用**代码知识树**结构，浅层加载到 Agent 记忆
- 左侧导航树对应项目目录结构 + 固定节点（架构概览、设计决策、开发约定）
- 每个节点初始只加载一两句摘要
- Agent 可调用 `expand_node(path)` 按需展开详细内容（函数签名、依赖关系等）
- 项目分析 Agent 巡检时只更新变更节点的摘要和内容
- 用户可手动编辑

## 5. 功能需求

### 5.1 项目管理系统

#### FR-001 多项目管理

- zero-core 支持关联多个项目目录
- 每个项目绑定一个独立的项目分析 Agent，工作目录即项目目录
- 一个项目分析 Agent 只管理一个项目，避免跨项目 Wiki 污染上下文
- 需求池跨项目共享，UI 可按项目筛选
- 首次关联项目时，项目分析 Agent 执行一次全量分析，生成完整 Wiki 浅层树（冷启动）

#### FR-002 定时巡检

- 项目分析 Agent 按可配置间隔（默认每天一次）自动运行
- **调度机制**：复用现有 Cron 系统（CronCreate），不做新调度器
- 巡检策略：增量 + Wiki 基线，不全量扫描
  - 读取 Wiki 快照作为知识基线
  - 只分析自上次巡检以来的增量变更（git diff、依赖变更等）
  - 用 diff 判断是否需要更新 Wiki 或创建需求
- 巡检内容：代码变更分析、依赖安全、测试覆盖率趋势、架构一致性、文档同步、功能演进建议
- 巡检结果自动生成需求卡片入池（状态：Found）
- 用户可随时手动触发巡检

#### FR-003 项目 Wiki 维护（代码知识树）

- Wiki 采用**代码知识树**结构，非传统长文
- 结构映射项目目录：每个文件/目录为一个节点，初始只含一两句摘要
- 固定顶层节点：架构概览、设计决策、开发约定
- **浅层加载**：Agent 初始化时只加载顶层树到上下文
- **按需展开**：Agent 调用 `expand_node(path)` 获取特定节点的详细内容（函数签名、依赖、调用关系）
- 存储在 SQLite 数据库中
- 每次任务归档后自动更新受影响节点的摘要和内容
- 用户可手动编辑

### 5.2 需求池系统

#### FR-004 需求状态机

- 完整的状态流转：Found → Discuss → Ready → Plan → Build → Verify → Closed
- 每次状态变更记录时间戳和触发者
- 异常流转：Verify 不通过 → Build 重做；任何状态可取消
- 状态历史完整可追溯

#### FR-005 需求重要等级

- 每个需求有重要等级字段：`low` / `normal` / `high` / `critical`
- 项目分析 Agent 自动评定重要等级
- 用户可手动调整等级
- 看板上按等级显示不同视觉标识（颜色/标记）

#### FR-006 智能通知

- Agent 自行判断内容重要性，分级通知
- **一般需求**（low/normal）：静默入看板，等用户自行选择是否处理
- **重要需求**（high/critical，如安全漏洞、架构风险）：入看板 + 在主聊天窗口主动通知用户去处理
- 通知内容包含：需求摘要、重要性原因、建议操作

#### FR-007 看板 UI

- 按状态分列展示需求卡片
- 卡片显示：标题、摘要、时间、执行状态、重要等级标记
- 支持按项目筛选（跨项目需求池共享）
- 筛选：按来源、优先级、项目、时间范围
- 搜索：全文搜索需求标题和描述

#### FR-008 需求对话

- **复用主聊天窗口**：点击看板需求卡片 → 跳转主聊天窗口 → 自动将该需求讨论作为用户消息输入
- 讨论阶段对话方为项目分析 Agent
- 执行阶段对话方切换为任务 Lead
- 对话历史随需求持久化

### 5.3 任务执行系统

#### FR-009 任务 Lead 调度（编排器架构）

- Lead 自动领取就绪（Ready）状态的需求
- Lead 使用**持续会话**的 AgentLoop，保持上下文连续性
- Lead 通过**编排器工具（orchestrate）**间接调度 sub-agent，不直接控制
- 编排器职责：创建 sub-agent AgentLoop 实例 → 注入上下文 → 执行 → 返回结果摘要
- Lead 指定 Wiki 节点和上下文传给编排器（MVP 阶段不做自动节点选择）
- 返回给 Lead 的是**摘要**（改了什么、审核了什么、有什么问题），非完整输出
- 提交规划等待审核（审核方可配置：项目分析 Agent 或用户）

#### FR-010 步骤执行

- 编排器为每个步骤创建**独立 AgentLoop 实例**（Dev/Reviewer/QA 各自独立）
- 注入的上下文：需求摘要 + Wiki 节点展开内容 + 具体任务指令
- 步骤按序执行，Lead 审核每步摘要结果
- 审核不通过可要求重做或调整方案
- 每个步骤可独立重试、可追溯

#### FR-011 角色模型配置

- 每个角色可配置使用的 LLM 模型
- 支持从已配置的 provider 中选择模型
- 默认按角色推荐模型分配
- 预置角色的 system prompt 允许用户直接编辑

### 5.4 验证与归档

#### FR-012 任务验证

- 执行完成后进入验证（Verify）状态
- 审核方可配置：项目分析 Agent 或用户
- 验证内容：变更是否符合原始需求和设计要求
- 验证不通过：回退到 Build 阶段重新执行
- 验证通过：进入归档

#### FR-013 归档与 Wiki 更新

- 验证通过后自动归档
- 项目分析 Agent 更新受影响的 Wiki 条目
- 生成任务完成报告：需求摘要、执行步骤、变更文件、测试结果

### 5.5 通用能力

#### FR-014 持久化与恢复

- 所有状态持久化到 SQLite
- App 重启或崩溃后，恢复所有需求状态和执行进度
- 项目分析 Agent 的定时巡检自动恢复

#### FR-015 失败处理

- 步骤失败：自动重试，默认最多 3 次
- 重试耗尽后升级：暂停执行，通知用户
- 用户可选择：重试 / 跳过 / 终止

#### FR-016 Git 集成

- 需求进入执行时自动创建 Git 分支
- 执行完成后可自动创建 PR
- PR 描述包含：需求摘要、执行步骤、变更文件列表

#### FR-017 并发控制

- 多个需求可同时处于不同状态
- 不做硬性并发限制，按系统资源自然调节
- 同一需求的步骤按序执行

## 6. 技术约束与决策

### 6.1 架构原则

| 原则 | 说明 |
|------|------|
| **复用现有 Runtime** | 每个 Agent Step 复用现有 AgentLoop，不另建执行引擎 |
| **Hook 驱动扩展** | 需求状态变更、步骤执行事件通过 Hook 暴露 |
| **本地优先** | MVP 仅支持本地执行，架构预留远程扩展点 |
| **持久化** | 复用现有 SQLite 存储层，新增需求/Wiki 相关表 |

### 6.2 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 需求状态机 | 自实现 | 状态流转明确，无需状态机库 |
| Agent 执行 | 复用 AgentLoop | 保持一致性 |
| Lead 调度 | 编排器作为 Lead 工具 | Lead 保有决策权，编排器屏蔽 sub-agent 管理复杂度 |
| Wiki 存储 | SQLite（代码知识树） | 树形结构 + 按需展开，与现有存储层统一 |
| 看板 UI | React + DnD | 与现有技术栈一致 |
| IPC 通信 | 扩展现有 IPC | 保持三层架构 |
| 定时巡检 | 复用现有 Cron | 不做新调度器 |

### 6.3 新增数据模型

## 9. 设计决策记录

### DD-001: 自驱动 vs 人工触发

**决策**：Agent 自驱动推进，用户只做方向把控和细节补充。

**理由**：软件开发中大量工作是重复性的（巡检、审查、测试），Agent 应主动承担。用户的价值在于方向判断和细节补充，而非逐一触发流程。

**影响**：需要可靠的定时调度 + 通知机制 + 清晰的状态展示。

### DD-002: 项目管理范围 (Q1)

**决策**：支持多项目管理，需求池跨项目共享，UI 可按项目筛选。

**理由**：用户通常同时维护多个项目，跨项目需求池避免信息孤岛。

### DD-003: Agent 协作架构 — 编排器模式 (Q2)

**决策**：编排器作为 Lead 的工具。Lead 调用 `orchestrate()` 控制执行步骤，不直接控制 sub-agent。编排器负责创建 AgentLoop 实例、注入上下文、执行、返回摘要。

**理由**：
- Lead 保有灵活决策权（根据上一步结果决定下一步做什么）
- Lead 的上下文只存摘要，不被 sub-agent 海量输出撑爆
- 每个步骤可独立重试、可追溯
- 编排器屏蔽了 sub-agent 管理复杂度

### DD-004: 项目分析上下文策略 (Q3)

**决策**：增量 + Wiki 基线。巡检时只看增量变更（git diff 等），不全量扫描。

**理由**：全量扫描 token 消耗巨大，增量策略配合 Wiki 基线已足够发现问题和更新知识。

### DD-005: Wiki 结构 — 代码知识树 (Q4)

**决策**：采用代码知识树结构（类似 code-graph-data.json），浅层加载 + 按需展开节点。Agent 可调用 `expand_node(path)` 获取特定节点详细内容。

**理由**：树形结构天然和代码目录对应，浅层加载节省 token，按需展开保证深度。

### DD-006: 通知机制 — Agent 自判重要性 (Q5)

**决策**：Agent 自行判断内容重要性分级通知。一般需求静默入看板，重要需求（安全漏洞等）同步在对话窗口通知用户。需求有重要等级字段。

**理由**：避免所有通知都打扰用户，但关键问题不能被淹没。

### DD-007: 需求对话 — 复用主聊天窗口 (Q6)

**决策**：复用现有主聊天窗口。点击看板需求卡片 → 跳转主聊天 → 自动将需求讨论作为用户消息输入。

**理由**：不引入新的对话面板，复用现有聊天基础设施，降低开发复杂度。

### DD-008: 定时巡检调度 — 复用现有 Cron (Q7)

**决策**：先复用现有 CronCreate/CronDelete 基础设施，不做新调度器。

**理由**：巡检本质是定时触发的 prompt，现有 Cron 足够。后续如需更复杂调度再优化。

### DD-009: 编排器返回摘要 (Q8)

**决策**：编排器返回给 Lead 的是结果摘要（改了什么、审核了什么、有什么问题），非完整输出。Wiki 节点由 Lead 在调用时指定，编排器不做自动选择。

**理由**：Lead 只需要决策信息，不需要看 Dev 的每一行代码。自动节点选择后续再做。

### DD-010: 冷启动 — 首次全量分析 (Q9)

**决策**：首次关联项目时，项目分析 Agent 执行一次全量分析，生成完整 Wiki 浅层树。

**理由**：一步到位，后续巡检才能在完整基线上做增量。

### DD-011: 项目管理 — 一项目一 Agent (Q10)

**决策**：一个项目分析 Agent 管理一个项目，工作目录即项目目录。不支持一个 Agent 管多个项目。

**理由**：多个项目的 Wiki 会污染和挤爆 Agent 上下文。一对一绑定保持上下文干净。
