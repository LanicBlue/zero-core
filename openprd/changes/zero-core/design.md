# 设计

## 背景

工具数量增加，误用情况变多，需要数据驱动优化。

## 目标

- 提供工具错误的详细统计和分析
- 基于错误上下文生成可执行的改进建议

## 范围

- 数据库：新增 tool_executions 表记录工具执行详情
- 记录层：在 agent-loop 捕获工具调用和错误
- 查询层：IPC 接口支持查询统计数据
- 前端：ToolsPage 新增统计 tab
- 分析：用户选择工具后，AI 基于错误上下文生成建议

## 约束

- 使用现有 SQLite 数据库
- 使用现有 LLM provider
- 不记录用户数据（用户 prompt 需脱敏）
- agent-loop 工具事件流
- session-db 数据库访问
- LLM provider

## 业务护栏

- 成本来源: LLM 分析调用产生 API 成本
- 额度限制: 每次分析消耗约 1k tokens
- 滥用防护: 限制分析频率（如每分钟 1 次）
- 监控信号: 分析调用次数、成功率
- 报警阈值: 分析失败率 > 20%
- 止损动作: 暂停分析功能，记录错误日志

## 风险与开放问题

- 假设: 错误记录足够多时，AI 能识别出模式
- 风险: AI 建议质量不可控
- 风险: 数据库写入可能影响工具执行性能
- 问题: 错误上下文需要记录到什么程度（用户 prompt 是否脱敏）？
